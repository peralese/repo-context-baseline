# repo-context-baseline

A small Python CLI that scans GitHub repositories, reads each repository README, and asks OpenAI to produce a standardized operational project baseline.

The goal is to create an external repository inventory that can later be compared against project context summaries generated by an OpenClaw Session Continuity Agent.

## What It Does

- Loads runtime settings from `config.json`
- Loads secrets from environment variables
- Authenticates to GitHub with `GITHUB_TOKEN`
- Lists repositories for a GitHub user or organization
- Fetches each repository README when available
- Optionally skips repositories without READMEs
- Sends README content to OpenAI with `OPENAI_API_KEY`
- Writes one markdown baseline per repository
- Writes a JSON repository inventory and a markdown baseline index for comparing repository, maturity, next step, and confidence
- Compares generated baselines against local OpenClaw `context.md` files
- Writes per-project comparison reports and an OpenClaw comparison index

## Requirements

- Python 3.11+
- A GitHub personal access token
- An OpenAI API key

## Setup

Create a virtual environment:

```bash
python3.11 -m venv .venv
source .venv/bin/activate
```

Install dependencies:

```bash
pip install -r requirements.txt
```

Create your local config and environment files:

```bash
cp config.example.json config.json
cp .env.example .env
```

Edit `.env` so it uses `KEY=value` lines:

```bash
GITHUB_TOKEN="your_github_token"
OPENAI_API_KEY="your_openai_api_key"
```

The CLI loads `.env` automatically if it exists. You can also export environment variables before running:

```bash
export GITHUB_TOKEN="your_github_token"
export OPENAI_API_KEY="your_openai_api_key"
```

## Configuration

Example `config.json`:

```json
{
  "github_owner": "YOUR_GITHUB_USERNAME",
  "owner_type": "user",
  "include_private": true,
  "include_archived": false,
  "include_forks": false,
  "include_missing_readme": false,
  "max_repos": 25,
  "openai_model": "gpt-4.1-mini"
}
```

Config fields:

- `github_owner`: GitHub username or organization name to scan
- `owner_type`: `user` or `org`
- `include_private`: include private repositories visible to your token
- `include_archived`: include archived repositories
- `include_forks`: include forked repositories
- `include_missing_readme`: generate a low-confidence baseline for repos without READMEs
- `max_repos`: maximum repositories to process
- `openai_model`: OpenAI model used for baseline generation

## Usage

Run the scan:

```bash
python src/main.py --config config.json
```

Limit the number of repositories:

```bash
python src/main.py --config config.json --limit 5
```

Process one repository by name:

```bash
python src/main.py --config config.json --repo REPO_NAME
```

Skip repositories that already have generated summaries:

```bash
python src/main.py --config config.json --limit 100 --skip-existing
```

Preview repository selection without calling OpenAI or writing generated summaries:

```bash
python src/main.py --config config.json --dry-run
```

Compare generated baselines with local OpenClaw project contexts:

```bash
python3 src/main.py --compare-openclaw
```

Compare one generated baseline with its matching OpenClaw context:

```bash
python3 src/main.py --compare-openclaw --repo openclaw-operator
```

The comparison command does not call GitHub or OpenAI. It reads existing files from
`outputs/summaries/` and looks for matching OpenClaw context files at:

```text
/Users/erickperales/Projects/<repo-name>/context.md
```

You can point it at a different local project root:

```bash
python3 src/main.py --compare-openclaw --projects-dir /path/to/Projects
```

The comparison generator can also be run directly:

```bash
python3 src/compare_openclaw.py
```

Or for one repository:

```bash
python3 src/compare_openclaw.py --repo openclaw-operator
```

Single-repository comparison runs write only `outputs/comparisons/<repo-name>.md` and
skip updating `outputs/openclaw-comparison-index.md`. Full comparison runs regenerate
the index.

## Outputs

Generated files are written to `outputs/`:

```text
outputs/
├── repos.json
├── baseline-index.md
├── openclaw-comparison-index.md
├── comparisons/
│   └── <repo-name>.md
└── summaries/
    └── <repo-name>.md
```

Example per-repository summary shape:

```markdown
# Project Baseline

## Repository
repo-name

## Repository URL
https://github.com/owner/repo-name

## Purpose
One or two sentences describing what the README says this project is for.

## Maturity
Functional MVP. The README describes a working CLI with documented configuration, outputs, and limitations.

## Current State
- CLI behavior is documented for full scans, repository-specific runs, dry runs, and skipped existing summaries.
- Generated outputs are written under `outputs/`.
- README-only summarization is the current evidence source.

## Key Capabilities
- Lists repositories for a configured GitHub user or organization.
- Fetches README content when available.
- Generates one baseline markdown file per repository.
- Writes a repository inventory and baseline index.

## Inferred Tech Stack
- Python 3.11+
- GitHub API
- OpenAI API
- Markdown and JSON outputs

## Operational Signals
- README documents CLI commands and configuration fields.
- README includes explicit limitations and roadmap items.

## Gaps / Risks
- Summaries rely only on README content, so source, issue, and pull request context may be missed.
- Dry runs do not validate OpenAI behavior.

## Primary Suggested Next Step
- Add comparison tooling against OpenClaw-generated context summaries.

## Additional Follow-Up
- Add tests around filtering, README handling, and writer output.

## Confidence / Notes
- Confidence: High
- The README provides direct usage, output, limitation, and roadmap evidence.
```

Example baseline index shape:

```markdown
# Baseline Repository Index

| Repository | Maturity | Primary Suggested Next Step | Confidence |
| --- | --- | --- | --- |
| [repo-name](summaries/repo-name.md) | Functional MVP. The README describes a working CLI with documented configuration, outputs, and limitations. | Add comparison tooling against OpenClaw-generated context summaries. | High |
```

Example OpenClaw comparison index shape:

```markdown
# OpenClaw Comparison Index

| Repository | Overall Assessment | Current State | Open Issues | Next Step | Recommended Fix |
| --- | --- | --- | --- | --- | --- |
| [repo-name](comparisons/repo-name.md) | Partially Aligned | Good | Partial | Good | No fix needed. |
```

Each comparison report deterministically extracts normalized markdown sections from the
baseline summary and the OpenClaw context. It maps equivalent headings such as
`Gaps / Risks` to `Open Issues`, `Primary Suggested Next Step` to `Next Step`, and
`Maturity` into current-state comparison. Missing OpenClaw contexts are listed clearly
in the comparison index and receive a short missing-context report.

## Limitations

- This MVP only uses README content, so summaries may miss important context from source files, issues, pull requests, or commit history.
- Very large READMEs are truncated before being sent to OpenAI.
- GitHub API rate limits and permissions depend on the token used.
- The app does not create a web UI, database, Docker setup, scheduler, or GitHub repository.
- Dry runs do not validate OpenAI behavior because they intentionally skip API calls.
- OpenClaw comparison is deterministic and markdown-based; it does not use an LLM evaluator yet.
- Comparison grading is a lightweight operational signal, not a semantic proof of correctness.

## Roadmap

- Add optional source tree metadata to improve operational context.
- Add optional LLM-assisted evaluation for OpenClaw comparison reports.
- Add resumable runs for large repository inventories.
- Add optional issue and pull request signal extraction.
- Add tests around filtering, README handling, and writer output.
